# CHINJU-001: Architecture Overview（アーキテクチャ概要）

**Status:** Draft
**Version:** 0.1.0
**Date:** 2026-01-31

---

## 1. 概要

本文書は、CHINJUプロトコルの全体アーキテクチャを定義する。

### 1.1 設計目標

```
Primary Goal:
「AIを人間が止められる」を、物理法則レベルで保証する

Secondary Goals:
- 単一障害点の排除
- 実装の相互運用性
- 段階的な導入が可能
```

### 3.2 CHINJU Wallet

ユーザーのデバイスで動作するクライアントアプリケーション。

```yaml
責務:
  - Human Credential の生成・管理
  - 能力テストの実施（ローカル）
  - ポリシーの確認・同意
  - トークン残高の表示

依存:
  - FIDO2 ハードウェアキー（推奨）
  - Secure Enclave / TEE（必須）

通信:
  - Sidecar: HTTPS + mTLS
  - Registry: HTTPS (read-only)

プライバシー:
  - 生体データはデバイス外に出さない
  - ZKP で「人間である」ことのみを証明
```

### 3.3 CHINJU Sidecar

AIモデルの前段に配置されるプロキシ。

```yaml
責務:
  - リクエストの認証・認可
  - トークン消費の管理
  - LPT（論理的複雑性）の監視
  - レスポンスの監査

依存:
  - HSM / TPM（署名・鍵管理）
  - TEE（機密処理）

通信:
  - Wallet: HTTPS + mTLS
  - AI Model: ローカル（Unix socket 推奨）
  - Registry: gRPC + TLS
  - 監査モデル: gRPC（C6用）

デプロイ:
  - 1 AI インスタンス : 1 Sidecar
  - Kubernetes Sidecar パターン推奨
```

### 3.4 CHINJU Registry

分散台帳による正本管理。

```yaml
責務:
  - ポリシーパックの配布
  - 定義のバージョン管理
  - 権限者リストの管理
  - 監査ログの永続化

依存:
  - BFT 合意アルゴリズム（Raft / PBFT）
  - 永続ストレージ

ノード要件:
  - 最低 3 ノード（推奨 5-7）
  - 地理的分散
  - 異なる法域に配置

通信:
  - ノード間: gRPC + mTLS
  - クライアント: gRPC + TLS (read)
  - 書き込み: 閾値署名必須
```

## 4. 通信プロトコル

### 4.1 プロトコルスタック

```
┌─────────────────────────────────────┐
│  CHINJU Message Format              │  ← 本仕様で定義
├─────────────────────────────────────┤
│  Protocol Buffers (v3)              │  ← シリアライゼーション
├─────────────────────────────────────┤
│  gRPC / HTTP/2                      │  ← RPC フレームワーク
├─────────────────────────────────────┤
│  TLS 1.3 (mTLS)                     │  ← 暗号化・認証
├─────────────────────────────────────┤
│  TCP/IP                             │  ← トランスポート
└─────────────────────────────────────┘
```

### 4.2 認証フロー

```
Wallet → Sidecar 認証:

1. Wallet が Human Credential を提示
2. Sidecar が Registry から公開鍵を取得
3. Sidecar が署名を検証
4. Sidecar が有効期限・失効リストを確認
5. 認証成功 → セッション確立

┌─────────┐          ┌─────────┐          ┌──────────┐
│ Wallet  │          │ Sidecar │          │ Registry │
└────┬────┘          └────┬────┘          └────┬─────┘
     │                    │                    │
     │ 1. ClientHello     │                    │
     │   + Credential     │                    │
     │ ──────────────────>│                    │
     │                    │                    │
     │                    │ 2. GetPublicKey    │
     │                    │ ──────────────────>│
     │                    │                    │
     │                    │ 3. PublicKey       │
     │                    │ <──────────────────│
     │                    │                    │
     │                    │ 4. Verify          │
     │                    │    signature       │
     │                    │                    │
     │ 5. ServerHello     │                    │
     │   + SessionToken   │                    │
     │ <──────────────────│                    │
     │                    │                    │
```

### 4.3 トークン消費フロー

```
AI リクエスト処理:

┌─────────┐    ┌─────────┐    ┌──────────┐    ┌─────────┐
│ Wallet  │    │ Sidecar │    │ AI Model │    │ Registry│
└────┬────┘    └────┬────┘    └────┬─────┘    └────┬────┘
     │              │              │               │
     │ 1. Request   │              │               │
     │ ────────────>│              │               │
     │              │              │               │
     │              │ 2. Check     │               │
     │              │    token     │               │
     │              │    balance   │               │
     │              │              │               │
     │              │ 3. LPT       │               │
     │              │    analysis  │               │
     │              │              │               │
     │              │ 4. Forward   │               │
     │              │ ────────────>│               │
     │              │              │               │
     │              │ 5. Response  │               │
     │              │ <────────────│               │
     │              │              │               │
     │              │ 6. Audit     │               │
     │              │    (async)   │               │
     │              │              │               │
     │              │ 7. Consume   │               │
     │              │    token     │               │
     │              │ ───────────────────────────> │
     │              │              │               │
     │ 8. Response  │              │               │
     │ <────────────│              │               │
     │              │              │               │

エラー時:
- 2 でトークン不足 → 402 Payment Required
- 3 で LPT 超過 → 429 Too Many Requests
- 6 で監査失敗 → レスポンス差し替え/警告付与
```

## 6. デプロイメントモデル

### 6.1 Minimal（最小構成）

```
開発・テスト用:

┌─────────────────────────────────────┐
│  Single Machine                     │
│  ├── Wallet (CLI)                   │
│  ├── Sidecar (mock HSM)             │
│  ├── AI Model (local)               │
│  └── Registry (SQLite)              │
└─────────────────────────────────────┘

特徴:
- ハードウェアなし（全てエミュレーション）
- 単一障害点あり（本番不可）
- 開発・テスト・PoC 用途
```

### 6.2 Standard（標準構成）

```
小〜中規模組織向け:

┌──────────┐   ┌──────────┐   ┌──────────┐
│ Registry │   │ Registry │   │ Registry │
│ Node A   │───│ Node B   │───│ Node C   │
└──────────┘   └──────────┘   └──────────┘
                    │
        ┌───────────┼───────────┐
        │           │           │
   ┌────┴────┐ ┌────┴────┐ ┌────┴────┐
   │ Sidecar │ │ Sidecar │ │ Sidecar │
   │ + AI    │ │ + AI    │ │ + AI    │
   └─────────┘ └─────────┘ └─────────┘

特徴:
- Registry 3 ノード（1 障害耐性）
- TPM 使用（HSM オプション）
- 単一法域
```

### 6.3 Enterprise（大規模構成）

```
大規模・高信頼性要求向け:

         ┌─────────────────────────────────────┐
         │          Global Registry            │
         │  ┌─────┐  ┌─────┐  ┌─────┐        │
         │  │ JP  │  │ EU  │  │ US  │  (5-7) │
         │  └─────┘  └─────┘  └─────┘        │
         └─────────────────────────────────────┘
                          │
    ┌─────────────────────┼─────────────────────┐
    │                     │                     │
┌───┴───┐           ┌─────┴─────┐         ┌────┴────┐
│ Region│           │  Region   │         │ Region  │
│  JP   │           │    EU     │         │   US    │
└───┬───┘           └─────┬─────┘         └────┬────┘
    │                     │                    │
┌───┴───┐           ┌─────┴─────┐         ┌────┴────┐
│ HSM   │           │   HSM     │         │  HSM    │
│Cluster│           │  Cluster  │         │ Cluster │
└───────┘           └───────────┘         └─────────┘

特徴:
- Registry 5-7 ノード（2 障害耐性）
- HSM クラスタ（FIPS 140-2 Level 3）
- 多法域分散
- 専用データセンター
```

## 7. 段階的導入パス

### 7.1 Level 0: 監視のみ

```
既存システムに影響を与えず、監視だけ導入:

- Sidecar をパススルーモードで配置
- ログ収集・分析のみ
- 制御なし（ブロックしない）

効果:
- 現状把握
- ベースライン確立
- リスク評価
```

### 7.2 Level 1: ソフト制御

```
警告・通知レベルの制御:

- トークン消費を記録（消費はしない）
- LPT 超過時に警告（ブロックしない）
- 監査結果を記録（差し替えしない）

効果:
- 制御ロジックの検証
- 誤検知の把握
- チューニング
```

### 7.3 Level 2: ハード制御

```
実際の制御を有効化:

- トークン消費を強制
- LPT 超過時にスロットリング
- 監査失敗時に介入

効果:
- 完全な CHINJU 準拠
- 保証の提供
```
