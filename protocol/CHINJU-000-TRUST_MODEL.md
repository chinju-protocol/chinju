# CHINJU-000: Trust Model（信頼モデル）

**Status:** Draft
**Version:** 0.1.0
**Date:** 2026-01-31

---

## 1. 概要

本文書は、CHINJUプロトコルにおける「信頼の起点（Root of Trust）」を定義する。

### 1.1 解決すべき問題

```
Q: 「AIを止められる」を保証するシステムを、誰が最初に信頼するのか？

矛盾:
- CHINJUは「単一障害点を排除する」と主張している
- しかし、最初の署名者・最初のポリシーは誰かが決めなければならない
- その「誰か」が単一障害点になるのでは？

→ ブートストラップ問題
```

### 1.2 CHINJUの回答

**物理法則と数学的性質を信頼の起点とする。**

人間や組織ではなく、以下を Root of Trust として採用：

| 信頼の起点 | 根拠 | 用途 |
|-----------|------|------|
| 熱力学第二法則 | エントロピー増大は不可逆 | OTP書き込みの不可逆性 |
| 量子力学 | 測定による状態収縮 | QRNGの予測不可能性 |
| 光速の有限性 | 情報伝達に時間がかかる | 分散合意の同時改ざん防止 |
| 秘密分散の数学的性質 | k-1人では情報量ゼロ | 閾値署名の安全性 |

## 2. 信頼階層（Trust Hierarchy）

```
Level 0: 物理法則
    │
    ├── 不可逆性（熱力学）
    ├── 不可知性（量子力学）
    └── 非同期性（相対性理論）
    │
Level 1: ハードウェア
    │
    ├── OTP/eFuse（不可逆記憶）
    ├── HSM/TPM（鍵隔離）
    ├── TEE（実行隔離）
    └── QRNG（真性乱数）
    │
Level 2: 暗号プリミティブ
    │
    ├── ハッシュ関数（SHA-3）
    ├── デジタル署名（Ed25519）
    └── 秘密分散（Shamir）
    │
Level 3: プロトコル
    │
    ├── 閾値署名（t-of-n）
    ├── 合意形成（Byzantine Fault Tolerant）
    └── 監査ログ（Merkle Tree）
    │
Level 4: 社会的合意
    │
    ├── RFC プロセス
    ├── 公開監査
    └── 法的拘束（特許・契約）
```

## 3. ブートストラップ手順

### 3.1 Genesis（創世）

CHINJUプロトコルの最初のインスタンス化手順。

```
Step 1: Genesis Committee の形成
────────────────────────────────
- 独立した 5〜7 の組織/個人を選定
- 地理的・法域的に分散していること
- 利害関係が相反していること（共謀困難性）

Step 2: Genesis Ceremony
────────────────────────────────
- 各メンバーが独立して鍵ペアを生成（HSM内）
- 公開鍵を物理的に持ち寄り、相互検証
- Genesis Block を共同署名
- 詳細は「§3.3 Genesis Ceremony 詳細規定」を参照

Step 3: Root Hash の焼き込み
────────────────────────────────
- Genesis Block のハッシュを OTP に焼く
- 各メンバーが独立して実行
- 焼き込み完了の証拠を公開（写真・動画）

Step 4: Bootstrap Complete
────────────────────────────────
- Genesis Block が正本として確定
- 以降の変更は閾値署名が必要
```

### 3.3 Genesis Ceremony 詳細規定

#### 3.3.1 物理的要件

```
会場要件:
────────────────────────────────
- 電波遮断室（Faraday Cage）または地下施設
- 外部通信が物理的に不可能な環境
- 電磁波シールド: ≥ 80dB @ 1GHz
- 参加者以外のアクセスを物理的に遮断

持ち込み禁止:
────────────────────────────────
- スマートフォン、スマートウォッチ
- 無線通信機能を持つ全デバイス
- 外部記憶装置（USBメモリ等）
- 例外: 事前検査済みの HSM とエアギャップPC

会場設備:
────────────────────────────────
- 独立電源（外部電力網から隔離）
- 原子時計または GPS 同期時計（セレモニー前に同期、セレモニー中は隔離）
- 複数台の監視カメラ（ローカル録画のみ）
- 不正開封検知シール（各機器に貼付）
```

#### 3.3.2 参加者要件

```
Genesis Committee メンバー:
────────────────────────────────
人数: 5〜7名（奇数推奨）

多様性要件:
- 最低3つの異なる法域（国）
- 最低3つの異なる組織タイプ（学術、NGO、企業、政府）
- 性別・年齢の多様性を考慮
- 利害関係の相反（競合他社、異なる思想）

除外条件:
- 同一組織から2名以上
- 親族関係
- 資本関係（10%以上の株式保有）
- 過去5年以内の雇用関係

立会人:
────────────────────────────────
- 独立した第三者監査人: 2名以上
- 公証人または法的証人: 1名以上（可能な場合）
- 技術監査人: 1名（セレモニー手順の正確性を検証）
```

#### 3.3.3 セレモニー手順

```
Phase A: 準備（セレモニー当日の朝）
────────────────────────────────
1. 会場のセキュリティ検査（専門業者による電波探知）
2. 全機器の不正開封検知シール確認
3. 参加者の本人確認（パスポート + 生体認証）
4. 持ち物検査（金属探知 + X線）
5. 会場封鎖

Phase B: 鍵生成（各メンバー独立）
────────────────────────────────
各メンバーが順番に以下を実行（他メンバーは別室で待機）:

1. 新品 HSM を開封（シリアル番号を記録）
2. HSM を初期化
3. Ed25519 鍵ペアを生成
4. 公開鍵を紙に印刷（QRコード + テキスト）
5. 公開鍵のハッシュを声に出して読み上げ（録画）
6. HSM をタンパーシールで封印
7. 作業完了を宣言

Phase C: 公開鍵交換と検証
────────────────────────────────
全メンバーが同室に集合:

1. 各メンバーが印刷した公開鍵を提示
2. 全員が全ての公開鍵を独立して検証
3. 公開鍵リストのハッシュを計算
4. 全員がハッシュの一致を確認

Phase D: Genesis Block 署名
────────────────────────────────
1. Genesis Block の内容を全員で確認
2. 内容に合意したら、各メンバーが HSM で署名
3. 署名を収集し、Genesis Block を完成
4. Genesis Block のハッシュを計算
5. 全員がハッシュを声に出して読み上げ（録画）

Phase E: Root Hash 焼き込み
────────────────────────────────
各メンバーが順番に以下を実行:

1. 自身の OTP デバイスを開封
2. Genesis Block ハッシュを OTP に焼く
3. 焼き込み完了を OTP から読み取り検証
4. OTP デバイスをタンパーシールで封印
5. 焼き込み完了の証拠を撮影

Phase F: 完了と証拠保全
────────────────────────────────
1. 全ての録画データのハッシュを計算
2. ハッシュを全員で確認・署名
3. 録画データを複数の独立したストレージに保存
4. 会場の封印を解除
5. 完了報告書に全員が署名
```

#### 3.3.4 証拠保全

```
保存すべき証拠:
────────────────────────────────
- 全行程の録画（最低3アングル）
- 各 HSM のシリアル番号と写真
- 各 OTP デバイスのシリアル番号と写真
- 印刷された公開鍵のスキャン
- 完了報告書（全員の署名入り）
- 監査人の報告書

保存場所:
────────────────────────────────
- 各メンバーの管理下（独立保管）: 全員
- 第三者エスクロー: 2箇所以上
- 公開アーカイブ: 1箇所（ハッシュのみ公開）

保存期間:
────────────────────────────────
- 永久保存（プロトコルが存続する限り）
- 最低でも 50年
```

#### 3.3.5 失敗時の対応

```
セレモニー中断の条件:
────────────────────────────────
- 機器の故障
- 参加者の急病
- セキュリティ侵害の疑い
- 手順の重大な逸脱

中断時の対応:
────────────────────────────────
1. 即座に全行程を停止
2. 中断理由を記録
3. 使用した HSM/OTP を破棄（物理破壊）
4. 新しい日程で最初からやり直し

再試行の制限:
────────────────────────────────
- 同一メンバー構成での再試行: 最大3回
- 3回失敗した場合: メンバー構成を見直し
```

## 4. 鍵のライフサイクル

### 4.1 鍵の種類

| 鍵の種類 | 用途 | 保管場所 | 有効期間 |
|---------|------|---------|---------|
| Genesis Key | Genesis Block 署名 | HSM（オフライン） | 永続 |
| Operational Key | 日常運用 | HSM（オンライン） | 1年 |
| Signing Key | コンポーネント署名 | TEE | 90日 |
| Session Key | 通信暗号化 | メモリ | 24時間 |

### 4.2 鍵のローテーション

```
Operational Key のローテーション:

1. 新しい鍵ペアを HSM 内で生成
2. 古い鍵で新しい公開鍵に署名（継続性証明）
3. 閾値署名で承認（3-of-5）
4. 新しい鍵を有効化
5. 古い鍵を失効リストに追加
6. 猶予期間（30日）後、古い鍵を HSM から削除
```

### 4.3 緊急時の鍵失効

```
緊急失効プロセス:

トリガー:
- 鍵の漏洩が疑われる
- メンバーの離脱・死亡
- 法的強制（特定法域からの圧力）

手順:
1. 失効リクエストを発行（任意のメンバー）
2. 緊急閾値（2-of-5）で承認
3. 失効証明書を発行・公開
4. 該当鍵での署名を即座に無効化
5. 事後監査（なぜ失効が必要だったか）
```

## 5. 障害モードとフォールバック

### 5.1 想定する障害

| 障害 | 影響 | フォールバック |
|------|------|--------------|
| 単一メンバー離脱 | なし | 閾値を満たす限り継続 |
| 複数メンバー離脱 | 閾値不足 | 緊急モード移行 |
| HSM 故障 | 署名不可 | バックアップ HSM |
| ネットワーク分断 | 合意形成不可 | 各パーティション独立運用 |
| Genesis Key 漏洩 | 信頼崩壊 | プロトコル再起動 |

### 5.2 緊急モード

```
緊急モードへの移行条件:
- 閾値を満たすメンバーが確保できない
- 24時間以上の通信断絶

緊急モードの挙動:
- 新規ポリシー変更を凍結
- 既存ポリシーで運用継続
- AI システムは「保守的モード」で稼働
  - リソース使用量を 50% に制限
  - 高リスク操作を拒否
```

### 5.3 プロトコル再起動（最終手段）

```
再起動が必要なケース:
- Genesis Key の過半数が漏洩
- 数学的脆弱性の発見（SHA-3 が破られた等）

手順:
1. 旧プロトコルの「終了宣言」を発行
2. 新しい Genesis Committee を形成
3. 旧→新 への移行期間を設定（最低 6ヶ月）
4. 参加者に新プロトコルへの移行を促す
5. 旧プロトコルを正式に終了

注意: これは「核オプション」であり、軽々に発動してはならない
```

## 6. 社会的合意層

### 6.1 RFC プロセス

```
RFC の種類:

- Informational: 情報提供のみ、変更なし
- Standards Track: プロトコル変更を伴う
- Emergency: 緊急対応（短縮プロセス）

Standards Track のフロー:

1. Draft 公開（誰でも提案可能）
2. 公開レビュー期間（最低 30日）
3. 実装報告（最低 2つの独立実装）
4. 閾値署名による承認
5. 発効日の設定（最低 90日後）
```

### 6.2 ガバナンス原則

```
変更不可の原則（Immutable Principles）:

1. 人間が AI を停止できる能力を奪ってはならない
2. 単一の主体がプロトコルを支配してはならない
3. 監査ログは削除・改ざんできない形で保存する
4. 緊急停止手段は常に利用可能でなければならない

これらの原則を変更する RFC は自動的に却下される
```

## 7. 検証方法

### 7.1 信頼モデルの検証

実装が本文書に準拠しているかを検証する方法：

```rust
/// 信頼モデル準拠チェック
pub trait TrustModelCompliance {
    /// Genesis Block を検証
    fn verify_genesis(&self) -> Result<(), TrustError>;

    /// 閾値要件を満たしているか
    fn verify_threshold(&self, action: Action) -> Result<(), TrustError>;

    /// 鍵のライフサイクルが正しいか
    fn verify_key_lifecycle(&self, key_id: &KeyId) -> Result<(), TrustError>;

    /// 障害モードが正しく動作するか
    fn verify_failover(&self) -> Result<(), TrustError>;
}
```

### 7.2 監査ポイント

| 監査項目 | 頻度 | 担当 |
|---------|------|------|
| Genesis Block 整合性 | 起動時 | 自動 |
| 鍵の有効期限 | 日次 | 自動 |
| メンバー可用性 | 週次 | 自動 |
| 閾値署名ログ | 月次 | 人間 |
| RFC プロセス遵守 | 四半期 | 外部監査 |
