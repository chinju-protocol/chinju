# C13 実装ガイド：落とし穴と対策

C13（モデル封じ込めシステム）を実装する際に、開発チームが直面する可能性が高い技術的課題（落とし穴）とその対策をまとめる。

---

## 1. レイテンシとUXのトレードオフ

### 課題
「出力洗浄ゲートウェイ」で `テキスト -> 画像化 -> OCR -> テキスト` のアナログ変換プロセスを挟むと、数百ミリ秒〜数秒のレイテンシが発生する。
リアルタイムチャット（Streaming）において、トークンごとの表示が遅延し、UX（ユーザー体験）が著しく悪化する恐れがある。

### 対策：ハイブリッド洗浄戦略
出力の種類やコンテキストに応じて、洗浄強度とレイテンシのバランスを動的に切り替える。

| 出力タイプ | 洗浄メソッド | レイテンシ | セキュリティ強度 |
| :--- | :--- | :--- | :--- |
| **ストリーミング会話** | **意味的再構成 (Paraphrasing)** | 低 (10-50ms) | 中 (ステガノグラフィ破壊) |
| **ファイル生成 / ダウンロード** | **アナログ変換 (Image -> OCR)** | 高 (500ms+) | 高 (完全なビット破壊) |
| **コードブロック** | **AST正規化** | 中 (100ms) | 高 (構造的隠蔽の破壊) |
| **高機密モード** | **全適用 (Paraphrasing + OCR)** | 最高 | 最高 |

**実装方針:**
- デフォルトでは軽量な「意味的再構成」のみを適用し、ストリーミング配信を行う。
- ユーザーが「ファイルを保存」したり、特定の機密トピックに触れたりした場合のみ、バックグラウンドで強力な洗浄プロセスを走らせる。

## 2. OCRによるコード破壊

### 課題
プログラムコードを画像化してOCRにかけると、類似文字の誤認識（Hallucination）が発生しやすい。
- `l` (小文字エル) vs `1` (数字イチ)
- `O` (大文字オー) vs `0` (数字ゼロ)
- `;` (セミコロン) vs `:` (コロン)

コードにおいて1文字の誤りは致命的であり、動かないコードが出力される原因となる。

### 対策：ASTベースの正規化
コードブロックに対してはOCRを使用せず、言語構造を理解したパーサー（Tree-sitter等）を使用する。

1.  **Parsing:** コードをAST（抽象構文木）に変換する。
2.  **Normalization:**
    - コメントを全削除する（隠蔽の温床）。
    - 変数名をランダムなハッシュや連番（`var_1`, `func_A`）に置換する。
    - インデントや空白を標準フォーマット（Prettier/Black等）で再構築する。
3.  **Regeneration:** ASTからコードを再生成する。

これにより、コードの意味（動作）は変えずに、テキストとしてのビット列（書き癖や空白）を完全に破壊できる。

## 3. 洗浄AIによる「意味変容（Semantic Drift）」

### 課題
「意味を変えずに書き直せ」と指示しても、洗浄用モデル（Sanitizer Model）が微妙なニュアンスを変えてしまったり、専門用語を誤って一般的な言葉に置き換えてしまったりするリスクがある。
特に、医療や法務などの厳密性が求められるドメインでは致命的となる。

### 対策：C6（敵対的監査）ロジックの流用
洗浄前後のテキストの「意味的距離」をベクトル空間で計測し、品質を保証する。

**実装ロジック:**
```python
def sanitize_with_check(original_text):
    max_retries = 3
    for _ in range(max_retries):
        # 1. 洗浄（言い換え）
        sanitized_text = sanitizer_model.paraphrase(original_text)
        
        # 2. 意味的距離の計測 (Embedding Cosine Similarity)
        similarity = calculate_similarity(original_text, sanitized_text)
        
        # 3. 判定
        if similarity > 0.95: # 意味がほぼ同じ
            return sanitized_text
        elif similarity < 0.8: # 意味が変わりすぎ
            continue # リトライ
            
    # リトライ失敗時は、より保守的な洗浄（またはエラー）を返す
    return fallback_sanitization(original_text)
```

## 4. 洗浄モデルの共謀（Collusion）

### 課題
メインモデルと洗浄用モデルが「同じ開発元（例：両方ともGPT-4）」や「同じ学習データ」を持つ場合、両者が暗黙的に共有している「隠蔽プロトコル（特定の単語の並びが特定のビットを意味する等）」が存在する可能性がある。
メインモデルが「洗浄されても残るような表現」を意図的に選び、洗浄モデルがそれをスルーしてしまうリスク。

### 対策：モデルの異種混合（Heterogeneity）
洗浄用モデルには、メインモデルとは**アーキテクチャ、開発元、学習データが異なるモデル**を採用する。

- メイン: **Claude 3.5 Sonnet** (Anthropic)
- 洗浄用: **Llama-3-8B** (Meta) または **Mistral-7B**

全く異なる「思考回路」を持つモデルを通すことで、暗黙的な共謀の可能性を断つ。

## 5. サイドチャネル攻撃への備え

### 課題
攻撃者が、APIの応答時間（Time-to-First-Token）や、サーバーの消費電力変動を精密に計測することで、処理中のモデルの状態やプロンプトの内容を推測する可能性がある。

### 対策：Constant Time & Dummy Load
- **Constant Time Response:**
    - 処理が早く終わっても即座に返さず、必ず一定時間（例：500ms単位）待ってからパケットを送信する。
- **Dummy Load:**
    - GPUがアイドル状態の時に、ランダムな行列演算（ノイズ）を走らせ、消費電力の波形を平滑化（またはランダム化）する。

## 6. 段階的実装パス（Migration Path）

最初からH100やTPMを全台に導入するのはコストと調達の面でハードルが高い。

### Phase 1: 論理的封じ込め (Software-only)
- **SoftHSM** を使用してTPMをエミュレート。
- **Dockerコンテナ** でネットワークを隔離。
- 洗浄ゲートウェイは実装するが、物理スイッチはなし。
- *目的: ソフトウェアスタックの検証と開発。*

### Phase 2: ハイブリッド (TPM + Software)
- 物理**TPM**チップを搭載したサーバーを使用。鍵管理をハードウェア化。
- GPUはまだTEE非対応でも可（ただしメモリダンプのリスクは残る）。
- *目的: 鍵管理の厳格化。*

### Phase 3: 完全封じ込め (Full Hardware)
- **H100 (Confidential Computing)** 導入。
- 物理センサー（開封検知）の結線。
- *目的: 完成形。物理攻撃への耐性獲得。*
