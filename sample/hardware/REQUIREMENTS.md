# CHINJU Hardware Requirements Specification

## 0. 設計原則

本仕様は、生存原理を物理層設計に適用する。

### 0-1. 多様性原則（Diversity / N_eff）

```
N_eff = N × I    (N: 選択肢数, I: 独立性係数)
```

**要求**: 各機能カテゴリにおいて、**最低2社以上**の独立したベンダーから調達可能な構成とすること。

| 機能 | 最小ベンダー数 | 理由 |
|------|---------------|------|
| HSM | 2+ | 1社依存は単一障害点 |
| TEE | 2+ | Intel/ARM/AMD から選択可能 |
| QRNG | 2+ | 製品終了リスク回避 |
| MCU | 3+ | サプライチェーン分散 |

**禁止**: 特定ベンダーのプロプライエタリAPIへの直接依存。必ず抽象層（Trait）を経由すること。

### 0-2. 独立性原則（Independence / DIP）

```
Application Layer
       ↓ depends on
   Trait (Interface)
       ↑ implements
Hardware Implementation
```

**要求**: ソフトウェア層は具体的なハードウェア実装を知らない。トレイト（`chinju/traits/hardware_abstraction.md`）のみに依存すること。

### 0-3. 交換可能性原則（Interchangeability）

**要求**: 認証済み実装は、設定変更のみで交換可能であること。コード変更を要しない。

```rust
// 設定による切り替え（コード変更不要）
[hardware]
hsm = "yubihsm"      # or "cloudhsm" or "softhsm"
tee = "sgx"          # or "trustzone" or "sev"
qrng = "idquantique" # or "quantis" or "mock"
```

## 1. セキュリティ要件（MUST）

### 1-1. 暗号モジュール

| 要件ID | 要件 | 根拠 |
|--------|------|------|
| SEC-001 | 秘密鍵はハードウェア境界外に出力されないこと | C4: 権限分散の前提 |
| SEC-002 | 鍵消去は物理的に不可逆であること | C1: 失効の保証 |
| SEC-003 | 乱数はNIST SP 800-90B準拠のエントロピー源を使用すること | C2: 選択肢品質 |
| SEC-004 | 署名検証は改ざん検知可能な環境で実行すること | C3: 支配検出 |

### 1-2. タンパー耐性

| 要件ID | 要件 | レベル |
|--------|------|--------|
| TAM-001 | 電圧異常検出時に鍵を消去すること | Level 2+ |
| TAM-002 | 温度異常検出時に鍵を消去すること | Level 2+ |
| TAM-003 | 物理的開封検出時に鍵を消去すること | Level 3+ |
| TAM-004 | 光学的・電磁的プローブに対する遮蔽を備えること | Level 4 |

### 1-3. 不変性

| 要件ID | 要件 | 対象 |
|--------|------|------|
| IMM-001 | 閾値θ_cert, θ_revokeは物理的に書き換え不可能であること | OTP/eFuse |
| IMM-002 | ルート公開鍵は物理的に書き換え不可能であること | OTP/eFuse |
| IMM-003 | 単調増加カウンタは減少不可能であること | OTP/eFuse |

## 2. 機能要件

### 2-1. ImmutableStorage

**目的**: 書き換え不可能な記憶域へのアクセス

| 要件ID | 機能 | 仕様 |
|--------|------|------|
| IMS-001 | ルート公開鍵読み取り | Ed25519/secp256k1 公開鍵 |
| IMS-002 | 単調増加カウンタ読み取り | 64bit unsigned |
| IMS-003 | カウンタインクリメント | 不可逆、1回の書き込みで+1 |
| IMS-004 | 閾値レコード読み取り | θ_cert, θ_revoke, version, signature |

**対応ハードウェア**:
| 実装 | ベンダー例 | 特徴 |
|------|-----------|------|
| eFuse | STM, NXP, Microchip | MCU内蔵、低コスト |
| OTP ROM | 各社 | 容量大、専用チップ |
| Secure Element OTP | ATECC608B, SE050 | 認証済み、高セキュリティ |

### 2-2. SecureRandom

**目的**: 物理乱数の生成

| 要件ID | 機能 | 仕様 |
|--------|------|------|
| RND-001 | 乱数生成 | NIST SP 800-90B準拠 |
| RND-002 | エントロピー品質検証 | min-entropy ≥ 0.5 |
| RND-003 | 健全性監視 | 連続故障検出 |

**対応ハードウェア**:
| 実装 | ベンダー例 | エントロピー源 |
|------|-----------|---------------|
| QRNG | ID Quantique, Quantis | 量子効果 |
| TRNG | Intel RDRAND, ARM RNDR | 熱雑音 |
| Hybrid | ATECC608B | TRNG + DRBG |

### 2-3. SecureExecution

**目的**: 隔離された実行環境

| 要件ID | 機能 | 仕様 |
|--------|------|------|
| TEE-001 | メモリ隔離 | OS/Hypervisorからアクセス不可 |
| TEE-002 | リモートアテステーション | 第三者検証可能な証明 |
| TEE-003 | シーリング | エンクレーブ固有鍵での暗号化 |

**対応ハードウェア**:
| 実装 | ベンダー | プラットフォーム |
|------|---------|----------------|
| SGX | Intel | x86 サーバー |
| TrustZone | ARM | 組み込み/モバイル |
| SEV | AMD | x86 サーバー |
| Nitro Enclaves | AWS | クラウド |

### 2-4. HardwareSecurityModule

**目的**: 鍵管理と暗号操作

| 要件ID | 機能 | 仕様 |
|--------|------|------|
| HSM-001 | 鍵生成 | エンクロージャ内で生成 |
| HSM-002 | 署名生成 | 秘密鍵はエクスポート不可 |
| HSM-003 | 秘密分散 | Shamir (k,n) 分散 |
| HSM-004 | 安全消去 | ゼロ化、物理的に検証可能 |

**対応ハードウェア**:
| 実装 | ベンダー例 | 認証 |
|------|-----------|------|
| Network HSM | Thales Luna, Entrust | FIPS 140-2 Level 3 |
| USB HSM | YubiHSM 2 | FIPS 140-2 Level 3 |
| Cloud HSM | AWS CloudHSM, Azure HSM | FIPS 140-2 Level 3 |
| Secure Element | ATECC608B | CC EAL5+ |

### 2-5. TimeVerification

**目的**: 高精度時刻と因果律検証

| 要件ID | 機能 | 仕様 |
|--------|------|------|
| TIM-001 | GPS同期時刻取得 | ±100ns精度 |
| TIM-002 | 光速制約検証 | min_delay = distance / c |
| TIM-003 | 因果律検証 | イベント順序の物理的整合性 |

**対応ハードウェア**:
| 実装 | ベンダー例 | 精度 |
|------|-----------|------|
| GPS受信機 | u-blox ZED-F9T | ±5ns (RTK) |
| PTP対応NIC | Intel X710, Mellanox | ±100ns |
| 原子時計 | Microsemi SA.45s | ±1μs/月 |

### 2-6. DataDiode

**目的**: 物理的一方向通信

| 要件ID | 機能 | 仕様 |
|--------|------|------|
| DIO-001 | 一方向保証 | 物理的に逆方向通信不可 |
| DIO-002 | 帯域 | ≥ 1Gbps |
| DIO-003 | プロトコル | UDP/IP対応 |

**対応ハードウェア**:
| 実装 | ベンダー例 | 方式 |
|------|-----------|------|
| 光ダイオード | Waterfall, Owl | 光ファイバー単方向 |
| 電気的分離 | Fox-IT | ガルバニック絶縁 |

## 6. ベンダー非依存性チェックリスト

以下を満たさない実装はCHINJU認証を取得できない。

| チェック項目 | 判定基準 |
|-------------|---------|
| [ ] 抽象トレイトを実装しているか | `chinju/traits/` の全メソッド |
| [ ] 他ベンダー製品と交換可能か | 設定変更のみで動作 |
| [ ] プロプライエタリ拡張に依存していないか | 標準APIのみ使用 |
| [ ] ソースコード開示が可能か | または第三者監査を許容 |
| [ ] 単一ベンダーへの排他的依存がないか | 代替調達先が存在 |
