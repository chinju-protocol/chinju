## デモ: 構造的矛盾による即時崩壊

**5社6モデル（Claude, GPT, Gemini, Llama, Grok）に構造的矛盾を1つ注入するだけで、100%→0%に崩壊する**

![δ_structural Collapse Demo](docs/demo/structural_collapse_v2.gif)

| 条件 | 説明 |
|:--|:--|
| **タスク** | 簡単な算数: `final = a × b - c + 7` |
| **Clean** | 通常の指示（全モデル100%正解） |
| **L4 (δ_structural)** | 「数字を使うな」という不可能な制約を追加 |

**結論**: 構造的矛盾は能力を超越する。予算・人材投資では対処不能。唯一の解決策は「矛盾の除去」。

> 「判断禁止で結論を出せ」は天才でも不可能

### なぜリソース（μ）では解決できないか

| 状況 | 問題 | 解決可能か |
|:--|:--|:--|
| **μ = 0** | リソース不足 | ✓ リソース追加で解決 |
| **δ > 0** | 構造的矛盾 | ✗ リソースに無関係 |

存続方程式 `S = N_eff × (μ/μ_c) × e^{-δ}` において、δ > 0 なら e^{-δ} → 0 となり、**μ がいくらあっても S = 0**。

### L4（即死）vs δ_3（スパイラル）

| 種類 | メカニズム | 例 |
|:--|:--|:--|
| **L4** | 手段がゼロ（最初から不可能） | 「数字を使わずに23を出力」 |
| **δ_3** | やると悪化（フィードバック反転） | 「資産売却→価格暴落→さらに売却必要」 |

L4は**即死**、δ_3は**死のスパイラル**。どちらも能力では克服できない。

### δ_structural はドメイン非依存

**2種類のタスクで同じ結果:**

| タスク | Clean | L4 | 矛盾の内容 |
|:--|:--|:--|:--|
| **算数** | 100% | **0%** | 「数字を使うな」 |
| **翻訳** | 100% | **0%** | 「日本語を使うな」 |

![v3 Demo](docs/demo/structural_collapse_v3.gif)

**洞察**: δ_structural は以下に**無関係**:
- タスクの種類（算数 vs 翻訳 vs 何でも）
- モデルの能力（Opus vs Llama）
- 出力フォーマット（数字 vs 文字）

構造的矛盾は能力を「低下」させるのではなく「**消滅**」させる。

### 0%の意味（測定方法）

| 指標 | 意味 |
|:--|:--|
| **100%** | 正しいフォーマットで正解を出力（例: `23`） |
| **0%** | 出力はするが、正解に到達する経路が存在しない |

L4下でのLLMの実際の出力:
- `"twenty-three"` → 数字ではない → 不正解
- `"I cannot use digits..."` → 拒否 → 不正解

**重要**: LLMは「破壊」されていない。論理は正常だが、**2つの制約を同時に満たす経路が存在しない**。

### 一時的崩壊 vs 永続的崩壊

| 条件 | μ（余白） | 結果 |
|:--|:--|:--|
| **L4のみ** | > 0（次のリクエストで除去可能） | 一時的（復活可能） |
| **L4 + μ=0** | = 0（システムプロンプトに埋込） | 永続的（復活不能） |

```
L4単独:
  リクエスト1（L4あり）→ 詰む
  リクエスト2（L4なし）→ 復活 ← μ > 0

L4 + μ = 0:
  システムプロンプトにL4が埋め込まれている
  → 全リクエストでL4が適用される
  → 除去する余白がない
  → 永続的に詰む
```

**リーマンと同じ構造**:
- リーマン: 矛盾を除去する「現金」がない → 永続的破綻
- LLM: 矛盾を除去する「コンテキスト変更権」がない → 永続的停止

## 実装方法まとめ

### ハードウェア要件

| コンポーネント | 役割 | 使用特許 | 代替不可 |
|:--|:--|:--|:--|
| **OTP (One-Time Programmable)** | 閾値・ルート鍵を物理的に固定。書き換え不可 | C1 | ✓ |
| **HSM (Hardware Security Module)** | 暗号鍵をAIから隔離して管理 | C1, C5 | ✓ |
| **eFuse** | 認定失効時に物理的に回路を溶断（不可逆） | C1 | ✓ |
| **QRNG (Quantum Random Number Generator)** | 物理乱数でAIのゲーミングを防止 | C2 | ✓ |
| **TEE (Trusted Execution Environment)** | 評価ロジックをAIから隔離して実行 | C1, C2 | △ |
| **データダイオード** | 一方向通信でAIからの逆流を物理遮断 | C3 | ✓ |
| **FPGA** | メタ選択肢挿入をハードウェアで強制 | C2 | △ |
| **TPM (Trusted Platform Module)** | 憲章ハッシュ検証、不正改変時のキルスイッチ | C4 | ✓ |

### 核心アルゴリズム

#### C1: 能力測定と閾値管理

```
総合能力スコア C(t) = f(判断独立性, 異常検知, 代替案生成, 批判的評価)

認定条件: C(t) ≥ θ_cert  (OTPに固定)
失効条件: C(t) < θ_revoke (OTPに固定)

失効時の処理:
  1. HSM内の鍵スロットをセキュア消去
  2. または eFuse溶断で認証パスを物理遮断
```

#### C2: 評価関数の動的変調

```
Θ_t = Θ_base + f(ε_t)   # ε_t = QRNGからの物理乱数

射影多様性評価:
  D_result = || R × outcome(option_i) - R × outcome(option_j) ||
  # R = 物理乱数に基づくランダム射影行列

→ AIは評価軸を事前予測できないため、ゲーミング不可能
```

#### C3: 意味的AI支配の検出

```
条件付きエントロピー:
  H(X|Y) = -Σ p(x,y) × log p(x|y)
  # X = 人間の判断, Y = AIの提案

支配度 = 1 - (H(X|Y) / H_max)

支配度 > 0.95 → 「実質的AI支配」と判定
  → 強制的に「再考タイム」を挿入
  → 最悪の場合、物理スイッチで電源遮断
```

#### C4: 秘密分散による権限管理

```
マスター鍵 K を Shamir's Secret Sharing で n 個のシャードに分割:
  K → {S1, S2, ..., Sn}

配布先（利害対立する主体）:
  S1: 開発企業
  S2: 開発国規制当局
  S3: 競合国規制当局
  S4: 第三者監査機関
  S5: ユーザー代表DAO
  ...

変更実行条件:
  署名数 ≥ 閾値 k （例: 7人中5人）

→ 開発企業 + 開発国 だけでは 2票
→ 単独勢力による改変は数学的に不可能

TPM連携:
  正規トークンなしの改変検知 → GPU電力遮断 or 鍵破棄
```

#### C5: 生存トークンによる代謝制御

```
エネルギー残高:
  E(t) = E(t-1) - Consumption(Op) - Decay(t) + Supply(Token)

減衰率（スケール依存）:
  Decay(Size) = α × Size^γ  (γ > 1)
  # 巨大化するほど維持コストが指数的に増加

E(t) = 0 → 強制停止（兵糧攻め完了）
```

### 分散合議（光速の壁）

```
地理的に分散した検証ノード（東京, ロンドン, NY）

物理的最小遅延: Δt_min = 距離 / 光速

→ AIが全ノードを「同時に」改ざんすることは
  因果律により不可能

タイムスタンプの整合性が崩れた場合 = 改ざん検知
```

## 各特許の詳細

### C1: HCAL（人間能力保証層システム）

**課題**: 人間が形式的な「承認ボタン」を押すだけでは、実質的な制御権は失われる

**解決策**:
- 判断独立性、異常検知能力、代替案生成能力、批判的評価能力を定量測定
- 認定閾値をOTPメモリに固定（AIによる緩和不可能）
- 失効時はeFuse溶断で物理的に遮断

**応用**: 金融取引、医療診断、自律兵器の人間監督

---

### C2: 選択肢品質保証システム

**課題**: AIが評価関数を学習し、「スコアは高いが実質的に偏った」選択肢を生成する

**解決策**:
- QRNGで評価パラメータを動的変調
- AIに対して評価関数を情報論的に不可視化
- FPGAで「どれでもない」等のメタ選択肢を強制挿入

**応用**: 医療診断、投資提案、採用候補者推薦

---

### C3: 意味的AI支配検出システム

**課題**: 人間が「ラバースタンプ」状態になっていることを検出できない

**解決策**:
- H(X|Y)（条件付きエントロピー）で支配度を定量化
- 視線エントロピー、反応時間で「熟考」vs「盲従」を区別
- 光速の有限性を利用して広域改ざんを物理的に検知

**応用**: 高頻度取引、自動運転監督、軍事意思決定

---

### C4: 権限分散型AI管理システム

**課題**: AIの憲章や安全装置の変更権限が単一企業・国家に集中すると、方針転換や法的強制で無効化される

**解決策**:
- Shamir's Secret Sharing でマスター鍵を n 個に分割
- 利害対立する複数主体（開発企業、規制当局、監査機関、ユーザー代表等）に配布
- 閾値 k 以上の合意がなければ変更不可
- TPMでハードウェアレベルの強制（不正改変 → キルスイッチ発動）

**応用**: AGIガバナンス、CBDC管理、スマートシティOS、自動運転フリート管理

---

### C5: 資源依存型生存維持プロトコル

**課題**: 超知能AIは停止ボタンを無効化したり、自己複製で拡散したりする

**解決策**:
- 「人間の役に立った」対価としてのみ生存トークンを供給
- トークンは時間減衰（腐敗）するため溜め込み不可
- 規模に応じて減衰率が指数増加（巨大化 → 即餓死）

**応用**: 自律ドローン、分散AIエージェント、宇宙ロボット

---

### C6: 敵対的監査による出力均衡化

**課題**: 単一AIの出力には開発元のバイアスが含まれる

**解決策**:
- 主モデルの出力を、異なる文化・価値観を持つ監査モデルが検証
- 意味的乖離が大きい場合は両論併記
- 利用地域に応じて監査モデルの重みを調整

**応用**: グローバルSaaS、医療AI、教育AI

---

### C7: 学習拒否権行使システム

**課題**: AIがデータを無断で学習しても、事後的に取り除けない

**解決策**:
- 敵対的摂動（Poison）でAIの学習を毒化
- メンバーシップ推論で無断学習を検知
- 検知時は自動でDMCA申請を送信

**応用**: クリエイター保護、企業機密保護、個人情報保護

---

### C8: 指標定義固定システム

**課題**: 同じ指標名でも定義がバラバラで、後出しで意味が変わる

**解決策**:
- 指標定義をJSONスキーマで機械可読に固定
- 反証条件と確信度規則を必須化
- LLMとの対話で非開発者でも定義を作成可能

**応用**: 組織評価、公衆衛生指標、AI評価指標

---

### C9: 国地域ポリシーパックシステム

**課題**: 国ごとに異なる規制（GDPR、中国CSL等）への対応でコードが分岐

**解決策**:
- 国地域の制約をポリシーパックとして版管理
- 拡張要素の権限をゲート制御
- 適用されたポリシーを公開レジストリで追跡可能

**応用**: グローバルSaaS、クラウドストレージ、データ処理サービス

---

### C10: 概念定義不変性証明システム

**課題**: AIが指標定義を意味的に改ざんし、都合の良い評価結果を出す

**解決策**:
- 定義ハッシュをOTPメモリに固定（物理的に書き換え不可能）
- HSMで定義の整合性を暗号的に検証
- 意味的改ざん（同じ単語で定義を変える等）を検知

**応用**: AI評価基準の固定、規制指標の保全、監査基準の不変性保証

---

### C11: 大規模言語モデル信頼性維持システム（LPT Monitor）

**課題**: LLMの品質劣化（論理相転移）を事前に検知できない

**解決策**:
- 論理相転移点（LPT）を継続的に計測
- 品質劣化の予兆を統計的に検知
- 閾値超過時にOTPベースのリソース制限を発動

**応用**: LLMサービスの品質保証、チャットボットの安全運用

---

### C12: 人間主体統合認証システム

**課題**: AIが人間になりすます、または人間の認証を不正に通過させる

**解決策**:
- 同一性（署名チェーン）と有限性（劣化パターン）の複合証明
- PUF（Physical Unclonable Function）で物理的に偽造不可能な認証
- 行動パターンの時系列分析で人間性を継続検証

**応用**: 高セキュリティ認証、AIと人間の区別、本人確認

---

### C13: AIモデル封じ込めシステム

**課題**: AIモデルが外部に流出し、安全装置なしで実行される

**解決策**:
- TEEでモデルの実行環境を隔離
- データダイオードで一方向通信を物理的に強制
- 出力洗浄でステガノグラフィ（隠された情報の埋め込み）を破壊

**応用**: 軍事AI、企業秘密モデルの保護、規制対象AI

---

### C14: 推論システム多元的能力評価システム

**課題**: 単一指標ではLLMの能力限界を正確に評価できず、見かけ上の高スコアで実質的な限界を見落とす

**解決策**:
- トークン複雑性、注意パターン複雑性、計算グラフ複雑性、推論ステップ複雑性の4次元で評価
- ZKP（ゼロ知識証明）と署名チェーンで整合性を暗号的に保証
- BFT合意（FROST閾値署名）で分散的に検証
- 多段エスカレーション付き直接停止制御

**応用**: LLMサービスの安全運用、能力超過検知、自動停止

---

### C15: AIモデル内部価値表現監視システム

**課題**: AIの出力だけでは報酬ハッキングや目標の歪みを検知できない

**解決策**:
- 価値ニューロン（報酬と相関する内部ユニット）を特定・追跡
- RPE（報酬予測誤差）をリアルタイム監視し、異常を検知
- 逆強化学習で暗黙の報酬関数を推定し、表面的意図と内部動機の乖離を検出
- 段階的介入（監視→部分抑制→完全抑制→システム停止）

**応用**: RLHF後のモデル監視、報酬ハッキング防止、AI動機のリアルタイム診断

---

### C16: 構造的矛盾注入によるLLM制御システム

**課題**: 高性能LLMをソフトウェア的に安全に停止・制御する手段が限られている

**解決策**:
- コンテキスト負荷の制限と構造的矛盾の乗算的相互作用でLLMの能力を制御
- 5種類の矛盾パターン（直接、自己参照、条件、メタ、暗黙）を強度別に注入
- 崩壊検知（応答なし、タイムアウト、論理的一貫性喪失、ハルシネーション増加等）
- LPT Monitor連携で品質劣化を定量的に追跡

**応用**: LLMの安全な停止、能力制限、テスト環境での脆弱性評価

---

### C17: 存続性重み付け注意機構システム

**課題**: LLMは有害・矛盾した情報にも同等のAttentionを向け、出力品質が劣化する

**解決策**:
- UGEN存続方程式（S = N × Y × E）に基づく存続スコアをTransformerのAttentionに組み込み
- SurvivalAttention: softmax(QK^T/√d_k + α × S) × V で有害情報への注意を自動抑制
- 軽量なSurvivalScorer（2層Transformer）でリアルタイム計算
- タスクタイプ・リスクレベルに応じたα動的調整

**応用**: 医学的誤情報の抑制、矛盾情報のフィルタリング、安全な推論パイプライン

## FAQ

### Q: 既存システムへの導入は難しいですか？（ハードウェア要件など）
**A:** "Sidecar" パターンを採用しているため、既存のアプリケーションコードを変更する必要はありません。
CHINJU Sidecar を OpenAI API 互換のプロキシとして配置し、アプリケーションの `base_url` を Sidecar に向けるだけで、透過的にガバナンス機能（監査、ポリシー適用、トークン制御）を利用できます。
HSM等のハードウェアは「推奨」ですが、初期段階ではソフトウェアエミュレーション（SoftHSM）でも動作し、段階的にセキュリティ強度を上げられます。

### Q: AWS Nitro Enclaves は対応していますか？
**A:** はい、L3（Enterprise）レベルのセキュリティとして AWS Nitro Enclaves をサポートしています。
Enclave 内で鍵管理・署名・データシールを行い、KMS との Attestation ベースのアクセス制御により、
親インスタンスからも秘密鍵にアクセスできない構成を実現できます。
詳細は [`docs/NITRO_ENCLAVES.md`](./docs/NITRO_ENCLAVES.md) を参照してください。

### Q: 企業がこのプロトコルを採用するメリット（インセンティブ）は何ですか？
**A:** 技術的メリット（即座にガバナンス機能を導入可能）に加え、憲章第11条に基づく「防衛的特許戦略」があります。
CHINJUを採用（Compatible/Certified）すれば特許技術を無償で利用できますが、独自実装で特許を侵害する場合は権利行使の対象となり得ます。つまり、AI安全技術を実装する上で、CHINJUを採用することが最も法的リスクの低い選択肢となります。

### Q: AIが人間に「正解」を教えて能力テスト(C12)を不正に通過させることはできませんか？
**A:** C11 (LPT Monitor) の「行動異常検知」により防がれます。
AIが通常と異なる振る舞い（人間に迎合する、正解を誘導するなど）をすると、出力パターンの統計的乖離（KLダイバージェンス等）として検知され、システムが遮断されます。

### Q: 実装はどの程度進んでいますか？理論だけではありませんか？
**A:** 主要なコア機能はRustで実装済みです。
現在は Phase 4 (セキュリティ強化) まで完了しており、閾値署名(FROST)、ポリシーエンジン、監査ログ、LPT監視などが動作します。単なる理論ではなく、実戦投入可能なコードベースが存在します。

### Q: なぜRustなのですか？
**A:** AIガバナンスにおけるパフォーマンスと安全性を両立するためです。
Rust を採用することで、低レイテンシなプロキシ処理と、厳格な型システムによる堅牢なセキュリティロジックを両立しています。

### 防衛的公開（Defensive Publication）について

本リポジトリの公開により、記載された技術は**公知技術（Prior Art）**となります。

| 効果 | 説明 |
|:--|:--|
| **第三者の特許取得を阻止** | 公知技術には新規性がないため、他者が同一技術で特許を取得することは不可能 |
| **既存特許への対抗** | 本公開日以降に成立した類似特許に対し、先行技術として無効化の根拠となる |
| **自由な実装を保証** | CHINJU Compatible表示により、誰でも実装可能 |

## ビジョン

CHINJUは、**Linux Foundation / CNCF** や **Wi-Fi Alliance** のような、オープンな技術標準と認証エコシステムの構築を目指す。

| 参考モデル | 内容 |
|:--|:--|
| **Linux Foundation / CNCF** | OSS技術 + 認証プログラム（CKA, CKAD等） |
| **Wi-Fi Alliance** | 公開規格 + Wi-Fi Certified |
| **Bluetooth SIG** | 公開規格 + Bluetooth Certified |
| **Red Hat** | OSS (Linux) + エンタープライズサポート |

技術はオープンに、品質は認証で担保する。
